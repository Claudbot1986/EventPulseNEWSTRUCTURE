### Canonical C-Pipeline Definitions (TARGET MODEL)

| Stage | Name | Role |
|-------|------|------|
| **C1** | Discovery / Frontier | Hitta och prioritera candidate pages. Samla interna links, mät page density, välj bästa kandidat. |
| **C2** | Grov HTML-screening + routing-signal | Screena HTML-struktur, detektera A/B/D-mönster, ge routing-signaler till testköer A, B eller D. Kan routea vidare till postTestC-A, postTestC-B, postTestC-D. |
| **C3** | HTML-extraktion | Extrahera events från HTML med regelbaserad logik (icke-AI). Är första platsen där verklig HTML-extraktion sker i målmodellen. |
| **C4-AI** | Separat AI-analys och/eller AI-fallback | Kör endast EFTER C1→C2→C3 har körts, och ENBART på fail-fallen. Lätt att koppla bort. |

**C4-AI = separat AI-analys-roll, INTE samma som 123-loopen.**

123-loopen är workflow-spec:en som styr hur förbättringscykler körs (Steg 0→1b→...→7).
C4-AI är en pipeline-stage: AI-analys som körs EFTER C1→C2→C3 och ENBART på fail-fall.

123 kan trigga C4-AI som en del av sin cykel, men de är olika saker.

**VIKTIGT: Två helt olika saker som båda heter "AI i C-spåret":**

| | C4-AI (förbättringsloop) | AI-som-fallback (dold extraktion) |
|---|---|---|
| **Syfte** | Analysera varför C0/C1/C2/C3 misslyckades, föreslå generella verktygsförbättringar | Extrahera events med AI när C3 misslyckas |
| **Körs på** | Alla fail-fall (samlat) | Enskilda fail-fall |
| **Output** | Förbättringsförslag, mönster, verktygsändringar | Events (fabricerade om C3 misslyckas) |
| **Är i C-kärnan?** | Nej — utanför pipeline | Ja — inbäddad i C-flödet |
| **Får användas i 123?** | Ja — enda tillåtna AI-användning | Nej — blockerad som "dold fallback" |

**Regel: I 123-loopen får AI endast användas för C4-AI-analys (förbättring av C0/C1/C2/C3 verktyg). AI-som-fallback för att "rädda" enskilda fail-fall är förbjuden.**

### Steg 0a: Sortera källor efter metod

Kör routinglogik för att sortera inkommande eller väntande källor:

| Metod | Villkor | Spår |
|-------|---------|------|
| **A — JSON-LD** | `triageResult = "jsonld_candidate"` ELLER JSON-LD upptäckt i preHtmlGate | A-spåret (ej C-loop) |
| **B — Network/API** | `triageResult = "network_candidate"` ELLER API/endpoints upptäckt | B-spåret (ej C-loop) |
| **C — HTML** | `triageResult = "html_candidate"` OCH ej bekräftad A/B/D | C-loopen |
| **D — Render/JS** | `pendingNextTool = "D-renderGate"` ELLER `likelyJsRendered = true` med hög konfidens ELLER `triageRecommendedPath = "render"` MED bevis (ej vag misstanke) | D-spåret (ej C-loop) |
| **E — Manual** | `triageRecommendedPath = "manual_review"` ELLER alla A→B→C→D testade och misslyckade | E-spåret (sista fallback) |

### Steg 0a-regler:

1. **C-batch-loop får endast innehålla C-källor** — bekräftade D-källor ska bort ur C-loopen omedelbart
2. **Vag D-misstanke ≠ bekräftad D** — källor med svag D-indikation kan fortfarande genomgå C-loop EN gång
3. **A och B går aldrig till C-loopen** — de har egna spår
4. **E är endast sista fallback** — när alla andra metoder bevisats otillräckliga

```

## STEG 0c: Definiera "Incoming C-Candidate Queue" (DOKUMENTATION)

**Detta är en formell definition, inte ett nytt steg.**

### Vad "incoming C-candidate queue" faktiskt är

Den "incoming C-candidate queue" som 123 loopen arbetar från består av:

1. **`batch-state.jsonl`** — huvudfilen som definierar:
   - `candidateQuery`: filterregler för C-kandidater
   - `batchSources`: exakt lista på 10 sourceIds valda enligt query
   - `currentBatch`: vilken batch vi är på
   - `status`: idle | testing | baseline_only | completed

2. **`runtime/sources_status.jsonl`** — källstatusfilen som förser batchen med data:
   - Varje sourceId i batchSources hämtas från denna fil
   - Fälten `triageResult`, `status`, `pendingNextTool`, `triageRecommendedPath` används för filtrering

### Formell definition

```
"incoming C-candidate queue" =
  batch-state.jsonl (batchSources + candidateQuery)
  ↓
  sources_status.jsonl (per-source data för batchSources)
```

### Viktigt: Var sparas grouping-metadata?

Grouping-metadata (`siteFamily`, `likelyCms`, `contentPatternGuess`, etc.) sätts av `phase1ToQueue.ts` men sparas för närvarande **inte** tillbaka till `sources_status.jsonl` permanent. Groupingdata finns endast i `phase1ToQueue.ts` `QueuedResult.grouping`-objektet under körning.

Detta innebär:
- Batch-byggandet görs via `candidateQuery`-filtrering, INTE via grouping
- Grouping är pre-C-information som `.deriveGrouping()` tar fram vid körning
- Nästa steg: grouping bör sparas till `sources_status.jsonl` under `grouping`-nyckeln

### 123 arbetar ENDAST från batchSources

123 loopens enda startpunkt är `batchSources` ur `batch-state.jsonl`. Den ska INTE:
- Hämta egna källor från `sources_status.jsonl`
- Bygga egna batchar från raw/import-underlag

**Nätverksfel-klassificering (OBLIGATORISK — NY) — C4-AI-måste köras innan slutsatser dras**

**Regel: 123 får ALDRIG behandla ett nätverksfel som automatisk routing-signal.**

Ett nätverksfel i fail-mängden måste FÖRST klassificeras som en av nedanstående nätverksfeltyper INNAN 123 får föreslå någon av följande:
- förbättra C0
- förbättra C2
- förbättra C3
- routea mot A
- routea mot B
- routea mot D
- lämna kvar som fail

C4-AI ska användas för att förstå nätverksfelets natur — inte för att gömma det eller förvandla det till en diffus slutsats.

**Obligatoriska nätverksfelkategorier:**

| Kod | Namn | Betydelse |
|-----|------|-----------|
| `url_problem` | Fel canonical URL | Fel domän, fel path-prefix, fel struktur — sourcen lever men nätverkslagret får fel svar |
| `dns_problem` | DNS misslyckades | Domänen går inte att slå upp — tillfälligt eller permanent |
| `timeout_problem` | Timeout | Fetch tog för lång tid — kan vara serverbelastning, nätverksproblem, eller för bred crawlnings |
| `tls_certificate_problem` | Certifikatfel | HTTPS-certifikatet är ogiltigt, utgånget, eller fel |
| `http_404_problem` | HTTP 404 | Resursen finns inte — kan betyda fel URL ELLER att sidan tagits bort |
| `http_403_problem` | HTTP 403 | Åtkomst nekad — kan betyda att sajten blockerar bots, fel IP, eller att sidan kräver auth |
| `http_5xx_problem` | Serverfel | Server-side error — tillfälligt eller permanent |
| `blocked_or_fetch_environment_problem` | Fetch-miljöproblem | Sajten blockerar headless-browsers, curl, eller liknande; annorlunda fetch-miljö kan behövas |
| `likely_requires_D` | Sannolikt D-signal | Nätverksfelet verkar bero på JS-rendering — content finns men kräver render |
| `likely_requires_A_or_B` | Sannolikt A/B-signal | Nätverksfelet verkar bero på att källan egentligen är API/network, inte HTML |
| `unclear_network_failure` | Oklart nätverksfel | Inget av ovanstående passar — kräver mer diagnostik |

**C4-AI nätverksfel-analys (obligatoriskt steg i 123-loopen):**

För varje fail källa där nätverksproblem observerades eller misstänks, ska C4-AI göra följande innan slutsatser dras:

1. **Konstatera nätverksfelet** — Vad sa fetch-felet? (t.ex. "DNS failure", "ECONREFUSED", "ETIMEDOUT", "certificate has expired")
2. **Klassificera nätverksfeltypen** — Vilken av ovanstående kategorier passar bäst?
3. **Avgöra orsaken** — Beror felet på:
   - fel canonical URL (source-config-problem)
   - verkligt DNS-problem (kan vara tillfälligt)
   - tillfällig timeout (kan vara serverbelastning)
   - cert-problem (kan vara konfigurationsfel på sajten)
   - fetch-miljöproblem (sajten blockerar viss typ av fetcher)
   - tydlig render-signal (content finns men kräver JS)
   - tydlig A/B-signal (sourcen är API/network, inte HTML)
   - eller något oklart
4. **Avgöra om routing är motiverad** — Om `likely_requires_D` eller `likely_requires_A_or_B`:
   - Routing föreslås ENDAST om nätverksfelet är tydlig och entydig signal, inte vag misstanke
   - Dokumentera varför routing föreslås
5. **Avgöra om verktygsförbättring är motiverad** — Om `timeout_problem` eller `blocked_or_fetch_environment_problem`:
   - Fel på en enskild sajt = site-specific → stoppa, använd source adapter
   - Fel på 2+ sajter med samma mönster = möjligen generellt → klassificera som "Provisionally General"
6. **Sätta confidence** — Hur säker är klassificeringen?

**Vad som INTE är tillåtet:**
- Att，自动iskt hoppa från "network error" till "kräver D" eller "kräver A/B"
- Att behandla alla nätverksfel som samma kategori
- Att använda nätverksfel som bevis för routing utan att först klassificera felet
- Att låta C4-AI gömma nätverksfel bakom en generell "fail"-kategori

**Exempel från batch-011 — korrekt hantering:**

❌ FEL slutsats (den gamla grova klassificeringen):

✅ KORREKT klassificering (exempel från batch-011):

### Steg 2b: Generalization Gate (OBLIGATORISK — SEPARAT STEG)

**INNAN du gör någon ändring i C-lager (C0/C1/C2), svara på dessa:**

1. **Vilken observation kommer från en enda sajt?**
   - Om ja → klassificera som "Site-Specific"
   - Om nej → fortsätt

2. **Finns samma mönster verifierat på minst 2–3 andra domäner?**
   - Om ja → klassificera som "General"
   - Om nej → klassificera som "Provisionally General"

3. **Är problemet egentligen canonicalization, parsing eller site naming?**
   - Canonicalization (www/www-less, trailing slashes) = ofta generellt
   - Parsing (DOM-struktur, selector-logik) = kan vara sitespecifikt
   - Site naming (företagsspecifika termer) = alltid sitespecifikt

4. **Riskerar ändringen att öppna brus på andra sajter?**
   - Att lägga till keywords ökar risken för brus
   - Att ta bort keywords minskar precision

**Klassificering och handling:**

| Klassificering | Handling |
|----------------|----------|
| **General** | Implementation tillåten i C-lager |
| **Provisionally General** | Gör INTE implementation ännu. Sök 2–3 andra domäner. Bekräfta först. |
| **Site-Specific** | STOPPA. Föreslå istället: source adapter, source-specific config, eller manual review |

**Hårda stopp-regler (gäller ALLTID):**
- IGNORE_PATTERNS får INTE ändras baserat på en enda sajt
- scoring-vikter får INTE ändras baserat på en enda sajt
- candidate ranking får INTE ändras baserat på en enda sajt
- URL token logic får INTE ändras baserat på en enda sajt
- negative keyword lists får INTE ändras baserat på en enda sajt

**Exempel:**
- "Folkoperan har `/pa-scen/arkiv/` som блокиras av 'arkiv' i IGNORE_PATTERNS" = Site-Specific → stopp, använd source adapter
- "Flera domäner har `www.example.com` vs `example.com` som olika origins" = General → normalization tillåten

Om svaret visar att rätt sida inte ens väljs ännu:
- ändra INTE extractor

### Render-Queue Blocking Rule (OBLIGATORISKT för 02-Ingestion)

**INNAN nästa steg väljs, kör denna kontroll:**

#### Steg A: Identifiera parkerade sources
1. Läs `{PROJEKT-ROT}/runtime/sources_status.jsonl` (eller `{PROJEKT-ROT}/02-Ingestion/sources_status.jsonl`)
2. Sök efter poster med `pendingNextTool: 'D-renderGate'` eller `status: 'pending_render'`
3. Dessa sources är **blockerade** - D-RenderGate finns inte ännu

#### Steg B: Klassificera aktuella candidates

| Evidens | Klassificering | Handling |
|---------|----------------|----------|
| C1 säger `likelyJsRendered=true` | **Verifierad render-kandidat** | Parkera omedelbart |
| C0 density hög MEN extraction=0 AND 403/redirect på event-paths | **Stark misstanke** | Parkera med `confidence: high` |
| Bara hög C0 density, ingen annan evidens | **Vag misstanke** | Tillåt fortsatt analys |
| HTML fungerar (events > 0) | **Ej render-kandidat** | Kan bearbetas nu |

#### Steg C: Beslutsregel

**OM** en källa är verifierad eller starkt misstänkt som render-kandidat:
- ✅ **GÖR:** Parkera i render-kö med tydlig status och reason
- ✅ **GÖR:** Sätt `pendingNextTool: 'D-renderGate'` och `confidence: high/medium/low`
- ✅ **GÖR:** Skriv reason, t.ex. `reason: "C0 density=113, extraction=0, /kalender/=403"`
- ❌ **GÖR INTE:** Rekommendera "undersök [källa]" igen
- ❌ **GÖR INTE:** Välj samma källa som huvud-nästasteg

**OM** render bara är vag misstanke:
- ✅ **TILLÅT:** Fortsatt analys om inget annat kan göras
- ✅ **MARKEA:** `confidence: low` tydligt

#### Steg D: Välj nästa steg från "runnable" sources

Nästa steg ska väljas från sources som:
1. Har bevisad eller möjlig HTML-path (events > 0 eller rimlig density)
2. Är INTE redan parkerade för D-renderGate
3. Kan köras med BEFINTLIGA verktyg (C0/C1/C2, extraction, scheduler)

**Priotiteringsordning:**
1. Sources med `events > 0` som behöver pipeline-verifiering (sourceTriage → phase1ToQueue)
2. Sources med `events = 0` MEN låg density OCH ej verifierad render → undersök
3. Sources med verifierad render → PARK och välj nästa

#### Exempel på fel beslut
```
❌ FEL: "Undersök malmolive" (redan 0 events, C0 density=113, /kalender/=403)
✅ RÄTT: "Kör sourceTriage på malmoopera" (events=8, bevisad path)
✅ RÄTT: "Parkera malmolive för D-renderGate, nästa steg: sourceTriage malmoopera"
```

Om en source ska parkeras för render, uppdatera `sources_status.jsonl`:

## Terminalkommando-regler (säkerhet)

### Undvik pipe-to-interpreter-mönster

**Förbjudet i 123:**
- `grep ... | python3` — triggar tirith `pipe_to_interpreter`
- `grep ... | node` — triggar tirith `pipe_to_interpreter`
- `grep ... | ruby` — triggar tirith `pipe_to_interpreter`
- `grep ... | perl` — triggar tirith `pipe_to_interpreter`
- `curl ... | python3` — triggar tirith `curl_pipe_shell` (även farligt)
- `cat ... | python3` — triggar tirith `pipe_to_interpreter`

**Tillåtna alternativ:**
- Kör Python/Node direkt med fil som argument: `python3 script.py` eller `node script.js`
- Filtrera filinnehåll med ren Python/Node: `python3 -c "import sys; print(open(sys.argv[1]).read())" file.txt`
- Använd `grep -E` för enkel filtrering (ingen pipe till interpreter)
- Använd `jq` direkt (tirith tillåter `grep | jq`)

**Bakgrund:** tirith (Hermes binära säkerhetsscanner) blockerar pipe-to-interpreter med `pipe_to_interpreter` (HIGH). Detta är designat för att stoppa attacker där skadlig kod hämtas och pipas till en interpreter. Vanliga utvecklingskommandon som `grep | python3` är falska positiver men kan inte skilljas från verkliga attacker utan att köra koden. Undvik mönstret helt i 123.
