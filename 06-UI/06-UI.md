# 06-UI

## Purpose

06-UI is the presentation layer of EventPulse. It is responsible for displaying real event data to the user in a clear, fast, and usable way. It is the final layer of the system — where truth meets the user.

## What belongs here

- UI components (screens, navigation, components, filters)
- Event list rendering
- Event detail views
- User interactions
- Loading, error, and empty states
- API client for fetching events from backend
- State management for UI

## What does NOT belong here

- Ingestion logic (belongs to `02-Ingestion`)
- Queue infrastructure (belongs to `03-Queue`)
- Normalization logic (belongs to `04-Normalizer`)
- Database schema (belongs to `05-Supabase`)
- Backend API server (lives in `services/api-server.cjs`)

## UI Flow (Ground Truth)

```
User opens app
    │
    ▼
Fetch events from backend (/supabase-events)
    │
    ▼
Render list view
    │
    ├── Loading state (while fetching)
    ├── Error state (if fetch fails)
    └── Empty state (if no events)
    │
    ▼
User can filter events
    │
    ▼
User taps event → navigate to detail page
    │
    ▼
Show full event data:
    title, date, venue, description, CTA
    │
    ▼
CTA ("See event") → open ticket_url
```

If any step breaks → user experience is broken.

## Data Responsibility

UI **must** display real data from backend — not fabricate or simulate events.

Rules:
- Show fallback UI when data is missing — not fake data
- Never hardcode sample events as if they were real
- Gracefully handle incomplete data
- Never hide backend errors from the user

## Rendering Rules

Every screen must have:
- **Loading state** — spinner or skeleton while fetching
- **Error state** — clear error message
- **Empty state** — "No events found"

Never show blank screens. Never show silent failures.

## Event List Rules

Must show: title, date, venue, key data.
Must be scrollable and responsive.
Must avoid: duplicate entries, empty rows, broken layouts.

## Event Detail Rules

Must include:
- title
- date/time
- venue
- description (if available)
- CTA ("See event") — **must work if link exists**

## Interaction Rules

- Respond instantly (optimistic UI where possible)
- Provide feedback (loading, pressed state)
- Never create dead buttons or unresponsive UI

## Forbidden Behaviors

- Fake events in UI
- Placeholder data presented as real
- Broken navigation
- Invisible errors
- Hardcoded data masking backend issues

## UI Loop Workflow

When improving UI, work in strict loops:

1. **Analyze** — What is broken? What is confusing? What does user see?
2. **Select ONE problem** — Never fix multiple UI issues at once
3. **Fix** — Smallest possible fix
4. **Verify** — Test in real UI, real user flow
5. **Evaluate** — Is it clearer? Did anything else break?
6. **Repeat** — Pick next problem

## Debug Rule

If UI is wrong, determine:
1. Is data missing from backend?
2. Is data fetched incorrectly?
3. Is data rendered incorrectly?

UI should not be blamed for backend issues.

## Relationship to Adjacent Layers

- **05-Supabase** — UI reads events via `api-server.cjs` `/supabase-events`
- **01-Sources** — events displayed originate from sources
- **02-Ingestion** — pipeline that processes events into storage

## AI Guidance

When working in UI:
- You may modify: components, screens, styles, navigation
- You must NOT: rewrite ingestion, redesign discovery, change database structure

UI quality comes from small, verified improvements. A clean, predictable, truthful interface builds trust.
